# 智学伴项目五人任务分工

## 1. 分工说明

本文档依据《智学伴系统业务设计》和当前代码完成情况，将剩余开发任务分配给 5 名成员。分工原则如下：

- 每人负责一个边界清楚、可以独立测试的业务域；
- 每个人都负责自己业务域的后端、数据表、测试和对应前端页面，避免只写接口不落页面；
- 最终整体联调、分支合并和答辩版本由组长统一负责，不单独占用一个成员名额；
- 公共代码修改前先在群内说明，避免多人同时修改 `backend/main.py`、数据库脚本和 `frontend/src/App.vue`；
- 所有接口必须从登录 Token 获取当前用户，不能相信前端传入的 `user_id`；
- 第一轮目标是完成答辩演示所需的完整业务闭环，不追求复杂算法和生产级商业能力。

## 2. 当前已有基础

以下内容已经具备基础实现，不需要重复开发：

- FastAPI、Vue 3、MySQL、Redis、Chroma 基础工程；
- 学生注册、登录、bcrypt 密码哈希、JWT 鉴权；
- Agent 基础调用与流式响应；
- Redis 短期记忆、MySQL 用户画像和 Chroma 长期记忆的基础代码；
- 登录注册、学习首页、智能学习、画像、记录、积分、会员和设置页面骨架；
- 统一异常响应、日志和健康检查基础能力。

当前 `learning_api`、`points_api`、`vip_api` 及相关 Service 目录大部分仍为空壳，是本轮开发重点。

## 3. 总体协作关系

```text
成员一：用户画像、学习计划、首次诊断、记忆生命周期
                         ↓ 提供学生画像
成员二：智能诊断、练习生成、答题分析
                         ↓ 产生答题结果
成员三：答案分析、掌握度、错题订正与复习
                         ↓ 形成学习结果
成员四：学习记录、首页推荐、学习报告与提醒
                         ↓ 触发权益消费
成员五：积分、使用次数、VIP 与支付宝沙箱
```

五名成员均需完成自己业务对应的 Vue 页面和接口接入。组长负责公共代码审核、最终合并以及全链路联调。

## 4. 成员一：用户画像、学习计划与记忆管理

### 4.1 负责范围

- 学生基础画像的创建、查询和修改；
- 学段、年级、学科或课程、学习目标配置；
- 每周学习天数、每日目标练习组数配置；
- 首次 5～10 道诊断题流程和跳过诊断逻辑；
- 根据首次诊断结果初始化知识点掌握度，跳过时初始化为 60；
- Redis 短期记忆生命周期；
- 用户切换会话、结束会话和清理短期记忆；
- 画像相关接口测试。
- 完成基础信息设置、学习计划和首次诊断对应的前端交互及接口接入。

### 4.2 需要实现的接口

| 方法 | 接口 | 说明 |
| --- | --- | --- |
| GET | `/profile/me` | 查询当前学生基础画像和学习计划 |
| PUT | `/profile/me` | 新增或修改当前学生画像 |
| GET | `/profile/diagnostic/status` | 查询是否需要首次诊断 |
| POST | `/profile/diagnostic/start` | 生成首次诊断题 |
| POST | `/profile/diagnostic/submit` | 提交首次诊断并初始化掌握度 |
| POST | `/profile/diagnostic/skip` | 跳过诊断并使用默认掌握度 |
| DELETE | `/sessions/{session_id}/memory` | 结束会话并清除短期记忆 |

### 4.3 数据任务

- 扩展 `user_profile`：学段、年级、学科、学习目标、每周学习天数、每日目标组数、是否完成首次诊断；
- 建立首次诊断记录及诊断答题数据；
- Redis 短期记忆设置 24 小时 TTL，每次新增对话刷新 TTL；
- 每个会话最多保留 10 轮，超出部分继续归档到长期记忆。

### 4.4 验收标准

- 新用户登录后会被引导完善画像；
- 可以完成或跳过首次诊断；
- 画像字段保存后重新登录仍能正确读取；
- 不允许用户修改或查询其他用户画像；
- Redis 会话键具有 TTL，24 小时无交互后能够自动过期；
- 提供接口测试和关键 Service 单元测试。
- 基础信息设置页不再使用固定演示数据，刷新后可以读取真实画像。

## 5. 成员二：智能诊断、练习生成与答题分析

### 5.1 负责范围

- 将现有 Agent 能力整理为稳定的学习业务接口；
- 识别用户意图和知识点；
- 结合成员一提供的学生画像生成薄弱点诊断；
- 每组生成 3～5 道针对性练习；
- 练习内容质量校验和一次自动重试；
- 简单、中等、困难三级难度调整；
- 答案判断、标准答案、基础解析和错因分类；
- 生成下一步学习建议；
- Agent 超时、生成失败和格式错误的降级处理。
- 完成智能学习页面中诊断结果、练习题和生成状态的前端展示。

### 5.2 需要实现的接口

| 方法 | 接口 | 说明 |
| --- | --- | --- |
| POST | `/learning/diagnose` | 输入题目、问题或薄弱点并返回诊断 |
| POST | `/learning/practices` | 根据诊断生成一组练习 |
| GET | `/learning/practices/{practice_id}` | 查询练习组和题目 |
| POST | `/learning/practices/{practice_id}/answers` | 提交答案并返回分析结果 |
| GET | `/learning/practices/{practice_id}/stream` | 可选的流式生成接口 |

### 5.3 输出数据约定

诊断结果至少包含：

```text
knowledge_point
mastery_score
learning_status
weakness
practice_suggestion
```

题目至少包含：

```text
question_id
content
difficulty
standard_answer
analysis
knowledge_point
```

答题反馈至少包含：

```text
is_correct
standard_answer
analysis
error_type
error_description
next_suggestion
```

### 5.4 验收标准

- 生成内容符合当前学段、年级和学科；
- 每组题目数量保持在 3～5 道；
- 题目、答案和解析不能为空，同组题目不能明显重复；
- 内容校验失败自动重试一次，仍失败时不创建有效练习记录；
- 错因只能归入知识点不会、计算错误、审题错误、方法错误之一；
- 连续答对两题提高难度，连续答错两题降低难度；
- 接口输出使用稳定的结构化 JSON，而不是让前端解析大段自然语言；
- 提供正常、模型失败、格式错误和重复提交测试。
- 智能学习页面能够展示真实结构化结果，不再只显示 Agent 原始文本。

## 6. 成员三：答案分析、掌握度、错题订正与复习

### 6.1 负责范围

- 知识点掌握度计算和状态更新；
- 答题记录和错题记录持久化；
- 错题解析、同类题巩固、订正流程；
- 订正后 1 天、3 天、7 天复习计划；
- 掌握度趋势和高频错因统计。
- 完成答题反馈、知识点画像、错题订正和到期复习对应的前端交互。

### 6.2 掌握度规则

| 答题结果 | 掌握度变化 |
| --- | ---: |
| 简单题答对 | +3 |
| 中等题答对 | +5 |
| 困难题答对 | +8 |
| 任意题答错 | -3 |

掌握度限制在 0～100，并同步映射为：

- 0～59：基础薄弱；
- 60～80：正在巩固；
- 81～100：掌握良好。

### 6.3 需要实现的接口

| 方法 | 接口 | 说明 |
| --- | --- | --- |
| GET | `/mastery/knowledge-points` | 查询知识点掌握情况 |
| GET | `/mastery/trend` | 查询掌握度变化趋势 |
| GET | `/mistakes` | 查询错题和订正状态 |
| POST | `/mistakes/{mistake_id}/correction` | 提交错题订正 |
| GET | `/mistakes/reviews/today` | 查询今日到期复习内容 |

### 6.4 验收标准

- 只有提交答案后才更新掌握度；
- 同一次答案提交不能重复更新掌握度；
- 答错后自动生成错题记录；
- 同一道错题首次订正成功后才能触发一次积分事件；
- 订正成功后正确生成 1、3、7 天复习时间；
- 提供掌握度边界、重复提交和复习日期测试。
- 学习画像页和错题订正页面展示真实数据并能完成订正操作。

## 7. 成员四：学习记录、首页推荐、学习报告与提醒

### 7.1 负责范围

- 学习会话和练习记录持久化；
- 历史学习时间线及按时间、学科、记录类型筛选；
- 首页主要推荐任务和次要推荐入口；
- 每日目标完成进度和站内学习提醒；
- 阶段性学情报告的数据统计与生成；
- 掌握度趋势、高频错因和下一阶段建议汇总；
- 完成学习首页、学习记录和阶段报告对应的前端页面及接口接入。

### 7.2 需要实现的接口

| 方法 | 接口 | 说明 |
| --- | --- | --- |
| GET | `/records` | 查询历史学习记录 |
| GET | `/recommendations/home` | 获取首页主要和次要推荐任务 |
| GET | `/plans/today` | 查询今日目标和完成进度 |
| GET | `/notifications` | 查询站内提醒 |
| POST | `/notifications/{notification_id}/read` | 标记站内提醒已读 |
| POST | `/reports/stage` | 生成阶段性学情报告 |
| GET | `/reports` | 查询历史报告 |
| GET | `/reports/{report_id}` | 查询报告详情 |

### 7.3 关键业务规则

- 推荐顺序为：今日到期错题 > 低掌握度且近期学过的知识点 > 学习目标相关内容；
- 首页只突出一个主要推荐动作，其他内容作为次要入口；
- 学习记录只能查看当前学生的数据；
- 每日目标进度根据有效练习和订正记录计算，不能由前端直接修改；
- 报告至少包含练习数量、正确率、掌握度变化、高频错因、薄弱点和下一步建议；
- 普通用户生成报告前由成员五的权益接口完成积分或 VIP 校验。

### 7.4 验收标准

- 学习记录能按时间、学科和记录类型筛选；
- 首页返回的首要任务符合推荐优先级；
- 每日目标完成后状态和进度正确更新；
- 报告统计结果能够与答题记录对应；
- 学习首页、记录页和报告页全部展示真实数据；
- 提供推荐顺序、统计结果、数据隔离和分页测试。

## 8. 成员五：积分、使用次数、VIP 与支付宝沙箱

### 8.1 负责范围

- 积分账户和积分流水；
- 登录、完善画像、有效练习、错题订正和连续学习奖励；
- 额外练习、详细错因分析和阶段报告积分兑换；
- 普通用户与 VIP 权益校验；
- 普通用户每日 5 组、VIP 每日 20 组的使用次数统计；
- 30 天 VIP 套餐及会员到期处理；
- 支付宝沙箱电脑网站支付；
- 支付异步通知、主动查询和订单幂等处理；
- 完成积分中心、会员中心、订单记录和支付结果页面的接口接入。

### 8.2 需要实现的接口

| 方法 | 接口 | 说明 |
| --- | --- | --- |
| GET | `/points/account` | 查询积分余额 |
| GET | `/points/transactions` | 查询积分流水 |
| GET | `/points/tasks` | 查询今日积分任务 |
| POST | `/points/exchanges` | 使用积分兑换功能 |
| GET | `/vip/status` | 查询 VIP 状态和到期时间 |
| GET | `/vip/usage` | 查询每日功能使用次数 |
| POST | `/vip/orders` | 创建 30 天 VIP 订单 |
| POST | `/vip/orders/{order_no}/alipay` | 创建支付宝沙箱支付 |
| POST | `/vip/alipay/notify` | 接收支付宝异步通知 |
| POST | `/vip/orders/{order_no}/query` | 主动查询支付宝交易 |
| GET | `/vip/orders/{order_no}` | 查询本地支付订单状态 |
| GET | `/vip/orders` | 查询当前学生订单记录 |

### 8.3 关键规则与验收标准

- 积分不能为负数，每次变化必须生成流水；
- 积分、奖励和次数使用唯一业务编号保证幂等；
- 只有练习成功生成后才扣减次数；
- 达到免费次数上限后正确提示积分兑换或开通 VIP；
- VIP 到期后自动按普通用户权益处理；
- 支付成功必须校验签名、APPID、订单号、金额和交易状态；
- 浏览器同步返回不能直接开通 VIP；
- 支付订单更新和 VIP 开通处于同一数据库事务；
- 同一订单只能增加一次 VIP 时长；
- 支付宝沙箱能完成创建订单、跳转支付、主动查询和开通 VIP；
- 支付取消、金额不符、签名错误和重复通知不会错误开通 VIP；
- 支付密钥只放 `.env`，不得提交到 Git；
- 积分和会员页面展示真实余额、流水、次数、订单及到期时间；
- 提供积分并发、次数边界、VIP 到期和支付幂等测试。

## 9. 公共接口约定

### 9.1 成功响应

```json
{
  "code": "SUCCESS",
  "message": "success",
  "data": {}
}
```

### 9.2 失败响应

```json
{
  "code": "BUSINESS_ERROR_CODE",
  "message": "可直接展示给学生的错误说明",
  "data": null
}
```

### 9.3 通用要求

- 时间统一使用 ISO 8601 格式；
- 数据库存储时间口径统一，前端按 Asia/Shanghai 展示；
- 分页接口统一使用 `page`、`page_size`；
- 写操作尽量携带 `request_id`，用于防止重复提交；
- 当前用户必须从 JWT 获取；
- 数据表必须带主键，关键幂等字段必须设置唯一索引；
- 新增接口必须补充 Pydantic 请求和响应模型；
- 新增业务规则必须有 Service 层测试，不能全部写在 API 路由中。

## 10. 推荐开发顺序

### 第一周：数据和接口骨架

- 成员一完成画像字段、学习计划和接口；
- 成员二确定诊断、练习、答案的 JSON 协议；
- 成员三建立答题、错题、掌握度和复习计划数据表；
- 成员四建立学习记录、推荐、提醒和报告数据表；
- 成员五建立积分、次数、VIP 和支付订单数据表；
- 每人同时建立自己业务域的前端 API 文件和页面数据结构。

### 第二周：核心学习闭环

- 成员一完成首次诊断和 Redis TTL；
- 成员二完成诊断、出题、判题和质量校验；
- 成员三完成答题反馈、掌握度、错题订正和复习计划；
- 成员四完成学习记录、首页推荐和每日目标；
- 成员五接入练习次数、练习积分和积分流水；
- 每人完成自己模块对应页面的第一轮接口接入。

### 第三周：商业化闭环

- 成员三完善错题复习和画像趋势；
- 成员四完成阶段报告、历史筛选和提醒；
- 成员五完成积分兑换、VIP、支付宝沙箱及对应页面；
- 组长组织第一次全链路联调，全员修复各自模块问题并补测试。

### 第四周：答辩准备

- 组长合并各成员分支并跑通完整演示流程；
- 准备固定演示账号和基础测试数据；
- 演练支付宝沙箱支付，并准备受配置控制的备用演示方案；
- 修复阻断演示的问题，不再临时扩展新功能；
- 统一接口文档、数据库脚本、README 和答辩说明。

## 11. 合并与联调规则

- 每人使用独立 Git 分支，例如 `feature/profile`、`feature/learning`；
- 一次提交只处理一个明确功能，不混入无关格式化；
- 数据库变更使用新增迁移脚本，不能直接修改线上数据或删除现有表；
- 修改公共响应格式、认证依赖或 `backend/main.py` 前先通知全组；
- 合并前必须通过本人模块测试和现有完整测试；
- 每名成员完成后端接口后，应立即接入自己负责的前端页面；
- 接口和页面不一致时先对照本文档，不在前端临时兼容多个随意格式；
- 组长负责公共接口审核、冲突处理、最终合并和整体联调；
- 每周由组长组织至少一次全链路联调，避免最后一周集中合并。

## 12. 最终共同验收流程

1. 注册学生账号并登录；
2. 完善学段、年级、学科、目标和学习计划；
3. 完成或跳过首次诊断；
4. 输入一个薄弱点并获得学情诊断；
5. 生成 3～5 道针对性练习；
6. 提交答案并展示正确答案、解析、错因和建议；
7. 更新掌握度并生成错题记录；
8. 完成错题订正并获得积分；
9. 查看画像、历史记录和首页推荐；
10. 验证普通用户每日 5 组限制；
11. 使用支付宝沙箱开通 30 天 VIP；
12. 验证 VIP 每日 20 组、详细分析、完整记录和报告权益；
13. 查看积分流水、支付订单和 VIP 到期时间；
14. 验证刷新页面、重新登录和重复提交不会造成数据错误。

只有以上流程能够从头到尾稳定完成，才视为本轮开发完成。
