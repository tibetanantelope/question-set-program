# 智学伴系统业务设计

## 1. 文档说明

本文档用于说明“智学伴”课程项目的业务定位、用户分类、核心学习流程、用户画像、积分与 VIP 机制等内容。

系统功能范围以现有《模块图》和《思维导图》为准，图中的模块与功能保持不变。考虑到本项目属于小学期课程项目，各功能采用简化、可实现的业务规则，不扩展为复杂的商业教育平台。

## 2. 项目定位

智学伴是一套面向小学、初中、高中和大学学生的智能学习辅助系统。学生可以输入不会的题目、学习问题或薄弱点，系统结合学生的基础信息和历史学习画像，完成知识点识别、薄弱点诊断、针对性练习生成、答案判断、错因分析和学习建议生成。

系统通过“输入问题—诊断学情—生成练习—提交答案—分析反馈—更新画像”的流程形成持续学习闭环，并采用“免费基础功能 + 积分激励 + VIP 订阅”的简化商业模式。

本项目的核心价值不是单纯使用大模型生成题目，而是根据不同学生的学习阶段和知识掌握情况，提供具有针对性的练习与反馈。

## 3. 目标用户

系统面向以下四类学生：

- 小学生
- 初中生
- 高中生
- 大学生

小学、初中和高中阶段默认使用人教版，不提供教材版本选择。大学阶段不设置教材版本，学生直接选择需要学习的课程，例如高等数学、大学英语或 Python 程序设计。

课程项目可以优先完整实现一个主要学科，例如数学；其他学科或大学课程保留业务入口和扩展能力。

## 4. 用户基础身份分类

### 4.1 基础身份信息

用户注册后首次使用系统时，需要完成基本学习信息设置。

| 信息 | 说明 |
| --- | --- |
| 学段 | 小学、初中、高中、大学 |
| 年级 | 根据学段选择对应年级 |
| 学科或课程 | 中小学选择学科，大学生选择课程 |
| 学习目标 | 日常巩固、薄弱点补习、考试复习 |
| 学习计划 | 每周学习天数、每日目标练习组数 |

教材版本不作为用户输入项。中小学统一按照人教版控制知识范围，大学课程根据课程名称和知识点组织学习内容。

学生首次选择学科或课程后，可以选择完成 5～10 道简短诊断题。系统根据诊断结果初始化相关知识点掌握度；学生跳过诊断时，相关知识点采用默认掌握度 60。首次诊断属于正式学习流程，用于提高初始画像和首次推荐的准确性。

### 4.2 学习状态分类

系统不对学生设置固定的“优秀学生”或“差生”标签，而是根据学生对不同知识点的掌握度，将其学习状态划分为三类：

| 掌握度 | 学习状态 | 业务含义 |
| ---: | --- | --- |
| 0～59 | 基础薄弱 | 对知识点理解不足，需要基础讲解和简单练习 |
| 60～80 | 正在巩固 | 已具备一定基础，需要继续练习和纠错 |
| 81～100 | 掌握良好 | 已基本掌握，可以尝试更高难度或综合练习 |

分类以知识点为单位。同一个学生在不同知识点上可以处于不同状态。例如，某学生“一元一次方程”的掌握状态为“正在巩固”，而“几何图形”的掌握状态可能为“基础薄弱”。

## 5. 用户角色

系统只面向学生用户，不设置管理员角色、管理员账号或管理后台。运行日志、异常排查和数据维护属于开发运维能力，不作为系统业务角色和前端功能。

### 5.1 学生用户

学生用户可以完成以下操作：

- 注册和登录系统；
- 设置学段、年级、学科或课程；
- 输入题目、问题或薄弱点；
- 获取学情诊断和针对性练习；
- 提交答案并查看反馈；
- 查看个人学习画像和历史记录；
- 完成任务获取积分；
- 使用积分兑换高级功能；
- 使用支付宝沙箱支付开通 VIP 并使用会员权益。

## 6. 核心学习闭环

系统的核心业务流程如下：

```text
选择学段、年级和学科（课程）
              ↓
输入题目、问题或薄弱点
              ↓
理解用户意图并抽取知识点
              ↓
结合用户画像诊断薄弱点
              ↓
生成针对性练习
              ↓
学生提交答案
              ↓
答案判断与错因分析
              ↓
更新知识点掌握度和学习画像
              ↓
展示反馈并生成下一步学习建议
```

### 6.1 学习内容输入

学生可以通过三种形式发起学习：

1. 输入一道不会的题目；
2. 描述一个学习问题；
3. 直接描述自己的薄弱点。

示例：

```text
题目输入：解方程 3(x-2)=12。
问题输入：移项时为什么要变号？
薄弱点描述：我做一元一次方程时经常在去括号和移项处出错。
```

系统需要对输入内容进行基础校验，包括是否为空、长度是否合理、是否属于学习相关内容等。

### 6.2 用户意图理解与知识点抽取

系统根据输入判断学生当前需要的服务类型，例如：

- 请求讲解题目；
- 请求分析薄弱点；
- 请求生成练习；
- 提交答案等待判断；
- 查询历史学习情况。

随后抽取相关知识点，作为诊断、练习生成和画像更新的依据。

### 6.3 薄弱点诊断

系统结合以下信息形成简化诊断：

- 用户的学段、年级和学科或课程；
- 用户本次输入的问题；
- 该知识点的历史掌握度；
- 相关历史错题和错因。

诊断结果至少包括：

```text
相关知识点
当前掌握度
学习状态
主要薄弱点
建议练习方向
```

示例：

```text
知识点：一元一次方程——移项
掌握度：55
学习状态：基础薄弱
主要问题：移项后符号处理错误
建议：先完成3道简单移项练习
```

### 6.4 针对性练习生成

系统根据诊断结果生成 3～5 道练习题。题目应符合用户当前学段、年级、学科或课程范围，并尽量围绕薄弱知识点生成。

练习生成后必须进行内容质量校验，包括题目是否完整、是否超出年级范围、答案与解析是否齐全以及题目是否重复。校验不通过时自动重新生成一次；再次失败则使用预置题或返回友好提示，不写入有效练习记录，也不消耗次数或积分。

题目难度只划分为三级：

- 简单
- 中等
- 困难

简化后的难度调整规则如下：

- 连续答对两题：下一题提高一个难度；
- 连续答错两题：下一题降低一个难度；
- 其他情况：保持当前难度；
- 难度不能低于“简单”或高于“困难”。

### 6.5 答案判断与错因分析

学生提交答案后，系统返回：

- 判断结果；
- 正确答案；
- 基础解析；
- 主要错因；
- 下一步建议。

课程项目中错因只划分为四类：

| 错因 | 说明 |
| --- | --- |
| 知识点不会 | 对概念、公式或规则不了解 |
| 计算错误 | 运算、符号或数值计算出现错误 |
| 审题错误 | 未正确理解题目条件或问题 |
| 方法错误 | 知道相关知识，但解题方法选择错误 |

示例反馈：

```text
判断结果：错误
正确答案：x=6
主要错因：计算错误
问题说明：移项后没有改变符号
学习建议：继续完成2道移项专项练习
```

### 6.6 学习建议生成

学习建议应简短、明确，并能够引导学生进入下一步操作。例如：

- 先查看基础概念，再完成简单练习；
- 继续完成两道同类题；
- 当前知识点已基本掌握，可以尝试中等难度；
- 建议重新练习历史错题。

首页和学习结束后的推荐统一按照以下优先级生成：到期复习的错题 > 掌握度较低且近期学过的知识点 > 与当前学习目标相关的内容。每次突出一个主要推荐动作，其他内容作为次要入口展示。

## 7. 用户画像设计

用户画像只保存与个性化学习直接相关的信息，不建立复杂的性格或行为标签体系。

### 7.1 基础画像

基础画像由用户设置，包括：

- 学段；
- 年级；
- 常用学科或课程；
- 学习目标。

### 7.2 知识点画像

每个知识点保存以下信息：

| 字段 | 说明 |
| --- | --- |
| 知识点名称 | 例如一元一次方程、函数极限 |
| 掌握度 | 0～100 的整数 |
| 学习状态 | 基础薄弱、正在巩固、掌握良好 |
| 答题次数 | 该知识点累计答题数量 |
| 正确次数 | 该知识点累计答对数量 |
| 最近学习时间 | 最近一次练习或更新的时间 |

### 7.3 掌握度更新规则

为了便于实现和答辩说明，掌握度采用规则计算：

| 答题结果 | 掌握度变化 |
| --- | ---: |
| 简单题答对 | +3 |
| 中等题答对 | +5 |
| 困难题答对 | +8 |
| 任意题目答错 | -3 |

新知识点的初始掌握度设为 60，最终掌握度限制在 0～100 之间。掌握度更新后，系统按照区间同步更新学习状态。

该规则主要用于课程项目演示，后期如果需要可以替换为更准确的学情评估算法。

### 7.4 历史错题

答错的题目进入历史错题记录，保存：

- 题目内容；
- 用户答案；
- 正确答案；
- 关联知识点；
- 错因；
- 答题时间；
- 是否已经订正。

错题需要形成“答错—查看解析—同类题巩固—完成订正”的完整闭环。学生完成订正后，系统更新订正状态并发放积分；订正通过的错题标记为“已掌握待复习”，并分别在 1 天、3 天、7 天后进入复习推荐。同一道错题只在首次完成订正时发放积分。

## 8. 记忆与数据业务

系统保留模块图中的短期对话记忆、长期学习画像、历史学习记录检索、知识点掌握度更新以及记忆提炼与归档功能，但采用简化分工。

| 数据类型 | 主要内容 | 建议存储位置 |
| --- | --- | --- |
| 短期对话记忆 | 当前会话的问题和上下文 | Redis |
| 用户与学习画像 | 基础信息、掌握度、积分、VIP | MySQL |
| 学习记录 | 练习、答案、错因、报告 | MySQL |
| 长期语义记忆 | 历史学习描述和相似问题检索 | Chroma |

记忆提炼与归档不需要设计复杂算法。一次练习结束后，系统将有价值的信息整理为知识点、掌握度、错因和建议，并更新到长期学习画像中即可。

## 9. 结果与记录

### 9.1 学情诊断展示

展示当前知识点、掌握度、学习状态、主要问题和练习建议。

### 9.2 练习题展示

展示题目内容、难度和答题区域。学生提交后显示判断结果。

### 9.3 答题反馈展示

展示正确与否、正确答案、基础解析、错因和下一步建议。

### 9.4 历史学习记录

记录每次学习的时间、学科或课程、知识点、题目数量、正确率和掌握度变化。

### 9.5 阶段性学情报告

阶段性报告可以按最近一段时间或最近若干次练习生成，内容包括：

- 完成练习数量；
- 总体正确率；
- 主要学习知识点；
- 掌握度变化；
- 高频错因；
- 当前薄弱点；
- 下一阶段建议。

普通用户可以使用积分兑换报告，VIP 用户可以直接生成报告。

## 10. 积分业务设计

积分用于鼓励学生持续学习，并为普通用户提供体验部分高级功能的途径。系统不设置积分等级、排行榜或复杂积分商城。

### 10.1 积分获取规则

| 行为 | 奖励积分 | 限制 |
| --- | ---: | --- |
| 首次完善个人学习信息 | +10 | 每个用户一次 |
| 每日首次登录 | +2 | 每日一次 |
| 完成一组有效练习 | +5 | 每日最多奖励三次 |
| 完成一道错题订正 | +3 | 同一道错题一次 |
| 连续学习三天 | +10 | 每个连续周期一次 |

“有效练习”指学生完成一组练习并提交答案，不能仅生成题目后退出。

### 10.2 积分消费规则

| 功能 | 消耗积分 |
| --- | ---: |
| 额外生成一组练习 | 10 |
| 查看一次详细错因分析 | 10 |
| 生成一次阶段性学习报告 | 20 |

积分不足时，系统提示用户继续完成学习任务或开通 VIP。

### 10.3 积分流水

每一次积分增加或扣除都必须生成流水，记录：

- 用户；
- 积分变化值；
- 变化原因；
- 变化后的余额；
- 发生时间。

积分发放和扣除由后端处理，前端只展示结果，避免重复领取或余额不一致。

## 11. VIP 业务设计

系统只设置普通用户和 VIP 用户两种会员状态，不设置会员等级。

### 11.1 VIP 套餐

课程项目只设置一种套餐：

```text
VIP会员：有效期30天
```

项目不接入生产支付环境，使用支付宝沙箱完成支付联调和课堂演示。用户在会员中心选择“支付宝沙箱支付”后，系统创建本地订单并跳转到支付宝沙箱收银台；只有后端确认沙箱交易成功后才能开通 VIP。沙箱交易不产生真实资金往来。

### 11.2 普通用户与 VIP 权益

| 功能 | 普通用户 | VIP 用户 |
| --- | --- | --- |
| 基础学情诊断 | 支持 | 支持 |
| 基础答案解析 | 支持 | 支持 |
| 每日练习生成 | 限制次数 | 较高次数或不限 |
| 详细错因分析 | 积分兑换 | 直接使用 |
| 历史学习记录 | 最近若干条 | 查看全部 |
| 阶段性学情报告 | 积分兑换 | 直接生成 |
| 个性化练习 | 基础生成 | 结合完整历史画像生成 |

为保证学生能够完成连续学习，普通用户每日可以免费生成 5 组练习，VIP 用户每日可以生成 20 组练习。每组仍为 3～5 题；额度差异用于控制大模型调用成本，不影响普通用户完成基础学习闭环。

### 11.3 VIP 权益校验

用户请求高级功能时，后端执行以下校验：

```text
判断VIP是否有效
      ↓
有效：检查使用次数并允许使用
      ↓
无效：判断用户是否选择积分兑换
      ↓
积分足够则扣除积分并允许使用
      ↓
否则提示开通VIP或获取积分
```

VIP 状态不能只由前端按钮控制，后端调用高级功能时必须再次校验。

### 11.4 支付宝沙箱订单

支付宝沙箱开通流程如下：

1. 用户选择 30 天 VIP，后端创建唯一的本地订单，初始状态为“待支付”；
2. 后端调用支付宝电脑网站支付接口生成支付表单，浏览器跳转到支付宝沙箱收银台；
3. 用户使用支付宝沙箱买家账号完成支付；
4. 支付完成后浏览器返回本地支付结果页，页面显示“支付结果确认中”；
5. 后端通过支付宝异步通知或主动交易查询确认真实交易状态；
6. 后端校验签名、应用编号、商户订单号、支付金额和交易状态；
7. 校验通过后，在同一数据库事务中将订单更新为“已支付”并开通 30 天 VIP；
8. 前端查询本地订单状态，展示支付结果和新的 VIP 到期时间。

本项目主要使用以下支付宝接口：

| 接口 | 用途 |
| --- | --- |
| `alipay.trade.page.pay` | 创建电脑网站支付并跳转沙箱收银台 |
| `alipay.trade.query` | 后端主动查询交易结果，支持本地演示 |
| 支付异步通知 | 支付宝服务器主动通知支付结果 |

本地地址可以作为浏览器同步返回地址；异步通知地址必须是支付宝服务器能够访问的公网 HTTPS 地址。开发阶段可以使用内网穿透提供临时通知地址。为避免课堂网络影响，系统同时保留主动查询机制，即使没有收到异步通知，也能由后端查询并确认支付结果。

订单状态简化为“待支付、支付中、已支付、已关闭、已退款”。第一轮必须实现前三种状态，关闭和退款可以保留接口扩展。支付通知和主动查询可能重复触发，因此订单更新及 VIP 开通必须具有幂等性，同一订单只能生效一次。

如果用户已经是有效 VIP，再次开通时在原到期时间上增加 30 天；如果 VIP 已到期，则从当前支付成功时间起计算 30 天。

支付宝应用私钥只能保存在后端环境变量或安全配置文件中，不得返回前端或提交到代码仓库。开发环境可以保留受配置开关控制的备用模拟支付入口，仅用于支付宝沙箱或课堂网络不可用时演示，默认关闭且不得出现在正式业务入口中。

## 12. 使用次数管理

系统对容易产生大模型调用成本的功能进行简单次数管理，包括：

- 每日练习生成次数；
- 每日详细错因分析次数；
- 每日学习报告生成次数。

次数在后端统一记录。不同会员状态使用不同上限，超过上限后提示用户等待次日、使用积分或开通 VIP。

次数口径统一如下：

- 普通用户每日免费练习 5 组，VIP 用户每日练习 20 组；
- 只有练习题成功生成并返回给学生后才扣减一次，生成失败、超时或内容校验不通过不扣次数；
- 每日次数按系统业务时区的自然日重置，前端展示“已使用 / 总额度”和剩余额度；
- 普通用户用积分兑换的额外练习不占免费额度，但仍计入当日总调用量；
- 同一请求使用唯一业务编号，重试不能重复扣减次数或积分。

本项目不实现复杂的并发限流，只实现按用户、按日期统计的业务次数限制。

## 14. 系统模块之间的业务关系

现有模块图中的七个模块保持不变，各模块关系如下：

```text
用户与画像模块
    ↓ 提供用户身份和历史画像
学习交互模块
    ↓ 接收题目、问题、薄弱点和答案
智能学习模块
    ↓ 完成诊断、出题、判题和建议
记忆管理模块
    ↓ 保存会话、记录并更新画像
结果与记录模块
    ↓ 向用户展示诊断、反馈和报告
会员与积分模块
    ↓ 控制高级功能、奖励和兑换

    └─ 为全部模块提供接口、认证、校验、流式返回、限流和日志能力
```

这七个模块不需要缩减。简化主要体现在分类数量、计算规则、会员套餐和订单流程上。

## 15. 关键业务规则

### 15.1 学习范围规则

- 中小学默认采用人教版；
- 中小学题目不能超出用户当前学段和年级范围；
- 大学生不设置教材版本，按照课程和知识点生成内容；
- 生成练习前必须获取用户的学段、年级和学科或课程信息。

### 15.2 画像更新规则

- 只有学生提交答案后才更新知识点掌握度；
- 单纯浏览或生成题目不改变掌握度；
- 掌握度必须限制在 0～100；
- 掌握度改变后同步更新学习状态；
- 答错题目应生成错题记录；
- 同一道错题只能获得一次订正积分。

### 15.3 积分规则

- 积分不能为负数；
- 积分获取与消费必须记录流水；
- 达到每日奖励上限后仍可学习，但不再发放对应积分；
- 高级功能只有在积分扣除成功后才能使用；
- 同一次业务请求不能重复发放或扣除积分。

### 15.4 VIP 规则

- VIP 以到期时间判断是否有效；
- VIP 到期后恢复普通用户权益；
- 已开通 VIP 的用户再次开通时顺延有效期；
- 高级功能必须由后端进行权益校验；
- 支付宝沙箱订单经后端确认已支付后才能开通 VIP；
- 同一支付订单只能开通一次 VIP，重复通知或重复查询不得重复顺延有效期；
- 前端同步返回结果不能直接作为支付成功依据；
- 支付金额、应用编号、商户订单号、签名和交易状态必须由后端校验。

## 16. 异常与边界处理

系统需要处理以下常见情况：

- 用户未登录时不能进入学习和会员功能；
- 用户未完善学段和年级时，先引导完善画像；
- 输入内容为空或长度超限时不调用大模型；
- 无法识别知识点时，提示用户补充学科和问题描述；
- 大模型生成失败时，返回友好提示并记录异常日志；
- 重复提交答案时，不重复更新掌握度和发放积分；
- 积分不足时，不执行兑换；
- VIP 已到期时，按普通用户规则处理；
- 普通用户达到使用次数上限时，提示使用积分或开通 VIP。
- 练习生成失败或内容校验不通过时，不扣减使用次数和积分；
- 用户取消支付或支付失败时，订单保持未支付状态且不开通 VIP；
- 支付宝异步通知暂未到达时，后端主动查询交易状态；
- 支付结果重复通知时，只返回处理结果，不重复更新订单或 VIP；
- 支付金额、签名或订单归属校验失败时，拒绝开通 VIP 并记录异常日志；

## 18. 课程项目实现优先级

### 18.1 第一阶段：核心学习闭环

必须完成：

1. 用户注册和登录；
2. 学段、年级、学科或课程及学习计划设置；
3. 首次简短诊断和初始画像建立；
4. 题目或薄弱点输入；
5. 知识点抽取和薄弱点诊断；
6. 针对性练习生成与内容质量校验；
7. 答案提交、判断和错因分析；
8. 掌握度及学习状态更新；
9. 错题解析、同类题巩固、订正和定期复习；
10. 推荐优先级、每日目标进度和站内提醒；
11. 历史学习和错题记录。

### 18.2 第二阶段：商业化闭环

必须完成：

1. 积分账户；
2. 积分获取和消费；
3. 积分流水；
4. 普通用户与 VIP 权益区分；
5. 每日使用次数管理；
6. 支付宝沙箱电脑网站支付；
7. 支付异步通知与主动交易查询；
8. 支付签名、金额、订单归属和交易状态校验；
9. 支付订单幂等处理与 30 天 VIP 开通；
10. 支付订单记录和支付结果页面。

### 18.3 第三阶段：展示优化

时间允许时完成：

- 阶段性学情报告；
- 掌握度可视化；
- 历史记录筛选；
- 流式结果展示。

## 19. 答辩演示流程

建议使用以下流程完整演示系统业务：

1. 注册一名初中七年级学生；
2. 选择数学和“薄弱点补习”；
3. 输入“我在一元一次方程移项时经常出错”；
4. 系统识别知识点并给出基础薄弱诊断；
5. 系统生成三道针对性练习；
6. 学生提交答案，其中一道故意答错；
7. 系统判断错因是计算错误，并给出建议；
8. 系统更新该知识点掌握度和学习状态；
9. 完成练习后获得积分并查看积分流水；
10. 普通用户尝试查看详细报告，系统提示积分兑换或开通 VIP；
11. 用户选择支付宝沙箱支付，系统创建待支付订单并跳转沙箱收银台；
12. 用户使用沙箱买家账号完成支付并返回系统；
13. 后端主动查询或接收通知，确认订单后开通 30 天 VIP；
14. 用户查看支付订单、VIP 到期时间、详细分析和阶段性学情报告。

该流程能够同时展示用户分类、学习画像、Agent 能力、学习闭环、积分激励和 VIP 商业模式。

## 20. 项目业务总结

智学伴通过学段、年级和学科或课程信息识别学生的基础学习身份，并根据不同知识点的掌握度，将学生在该知识点上的状态划分为“基础薄弱、正在巩固、掌握良好”。系统围绕学生输入的问题生成针对性练习，在学生答题后完成判题、错因分析和画像更新，从而形成持续的个性化学习闭环。

在商业化方面，系统保证普通用户可以正常使用基础学习功能，通过积分鼓励持续学习并兑换部分高级能力，同时由 VIP 提供更多练习次数、详细错因分析、完整历史记录和阶段性学习报告。

该设计保留了现有模块图和思维导图中的全部核心功能，同时通过减少分类数量、固定中小学教材版本、简化掌握度算法、设置单一 VIP 套餐并接入支付宝沙箱，使项目规模适合课程开发与答辩展示。
